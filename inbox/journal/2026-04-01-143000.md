Had an interesting idea during lunch about API error handling. We've been returning generic error objects with status codes and messages, but what if we structured errors as a discriminated union with machine-readable error codes?

Instead of `{ status: 400, message: "Invalid email format" }`, something like `{ error: "VALIDATION_FAILED", field: "email", constraint: "format", detail: "Expected RFC 5322 format" }`. The client can then pattern-match on the error code and field to show localized, context-specific messages.

Alex mentioned something similar in last week's review -- they called it "typed errors." The key insight is that the error response should be as structured as the success response. We already version our success schemas, why not version error schemas too?

For the API v2 migration this could be a big win. If we define error types upfront, client teams can build proper error handling without reverse-engineering our error strings.
