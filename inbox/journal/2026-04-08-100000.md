Kicking off the LLM eval harness side project this morning. Spent an hour sketching before I touched any code.

The hardest question is what to actually measure. My first instinct was "response quality" but that's a black box that means nothing. Breaking it down:

- **Response quality**: Needs a rubric. For the error explainer, something like: does it name the root cause? does it suggest a fix? is the fix correct for the given stack trace? Each is a yes/no an LLM judge can score.
- **Latency**: Easy to capture, easy to trend. p50, p95, p99 per prompt version.
- **Cost per eval**: Token counts in and out, times the model's price. Non-negotiable — we need to know when a "better" prompt is actually 3x more expensive.
- **False positives / false negatives on code review suggestions**: This is the copilot feature. Needs a labeled golden set. Smaller than I'd like at first, but can grow.

I'm deliberately not going to build dashboards, auth, a server, or anything SaaS-shaped. CLI in, Markdown out, commit the results to a repo. If someone wants a dashboard later, they can grep the Markdown.

Starting with the error explainer. It has the clearest rubric and the best test cases sitting in our bug tracker already.
