## Typed Error Responses for API v2

Idea for replacing generic HTTP error responses with structured, machine-readable error types using a discriminated union pattern.

### Current Approach (v1)

```json
{ "status": 400, "message": "Invalid email format" }
```

Clients must parse human-readable strings to determine error type. Fragile and impossible to localize.

### Proposed Approach (v2)

```json
{
  "error": "VALIDATION_FAILED",
  "field": "email",
  "constraint": "format",
  "detail": "Expected RFC 5322 format"
}
```

Clients pattern-match on structured fields. Error schemas are versioned alongside success schemas.

### Key Insight

Error responses deserve the same schema discipline as success responses. Alex Chen called this "typed errors" during a recent review. Defining error types upfront enables client teams to build proper error handling without reverse-engineering error strings.

### Relevance to API v2

This could be a significant improvement in the v2 migration. If error types are part of the API contract, breaking changes in error handling become visible and manageable.
