Starting to think more concretely about the API v2 migration strategy. The biggest risk is the transition period where we need to support both v1 and v2 simultaneously. We have about 15 active clients on v1.

Options I see:

1. **Big bang**: Set a sunset date, migrate everyone at once. Risky but clean. Probably 3-month timeline.
2. **Parallel operation**: Run both versions indefinitely. Safe but doubles maintenance burden.
3. **Incremental endpoint migration**: Move endpoints one by one to v2 semantics. Clients can adopt gradually. Need a good versioning strategy at the endpoint level.

I'm leaning toward option 3. We could use content negotiation (Accept header with version) or path-based versioning (/v2/resources). Path-based is more explicit and easier for clients to understand, even if it's less RESTful.

Need to discuss this with Alex and get their take on the infrastructure implications. The database schema changes required for v2's new entity model could be the biggest blocker.
